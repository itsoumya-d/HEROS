# Forge v0.2 Design Spec

**Status:** DRAFT — Cycle 37 (2026-05-19)
**Author:** Red team / perpetual loop
**Blocks:** V39 (decision_required non-enforcement), V34 (Zero stdin gap)
**Depends on:** Zero v0.2 (world.in), std.fs.rename

---

## 1. Problem Statement

### V39 — decision_required is advisory only in v0.1

`forge_analyze` returns `"decision_required": true` when a migration has `has_data_loss: true` or `risk_tier: CRITICAL`. The MCP manifest instructs agents:

> "If true, agent MUST NOT auto-proceed. Surface to human principal before executing migration."

But there is **no enforcement**. An autonomous agent can read `decision_required: true` and proceed anyway. The forge bridge has no way to:
1. Gate a subsequent "execute" call behind human approval
2. Verify that a human principal actually reviewed the risk analysis
3. Differentiate "agent proceeded with human approval" from "agent ignored the flag"

This gap is LLM06 (Excessive Agency) in OWASP LLM Top 10 (2025) — the tool advertises a safety contract it cannot enforce.

### V34 — forge binary cannot read stdin

Zero v0.1.x has no `world.in`. The bridge passes schemas as CLI arguments. This creates a 64 KiB practical ceiling on schemas (shell argument length limits). v0.2 replaces the bridge with a native Zero MCP server once `world.in` is available.

## 2. V39 Fix: human_acknowledgment_token

### 2.1 Design

A new optional parameter `human_acknowledgment_token` is added to `forge_analyze`. When the analysis result has `decision_required: true`, agents MUST provide this token on a second call to receive a `proceed_ok: true` authorization — without which any downstream execution tool (not yet built) can verify authorization was granted.

The token is a **nonce** generated by the forge bridge at the point where the human was asked. The flow:

```
Agent calls forge_analyze(from, to)
  → Bridge detects decision_required: true in response
  → Bridge appends: "approval_nonce": "<16-hex-char nonce>"
  → Agent surfaces analysis + nonce to human principal
  → Human says "proceed"
  → Agent calls forge_analyze(from, to, human_acknowledgment_token: "<nonce>")
  → Bridge verifies nonce matches a pending approval record
  → Response now includes: "proceed_ok": true, "decision_required": false
  → Agent may now call a downstream executor
```

### 2.2 Wire Format

#### First call (no token) — decision_required case

Input:
```json
{ "from_schema": "...", "to_schema": "..." }
```

Output (additional fields when decision_required: true):
```json
{
  "risk_tier": "CRITICAL",
  "has_data_loss": true,
  "decision_required": true,
  "approval_nonce": "a3f7e2c1b8d04591",
  "approval_prompt": "Migration has data loss. Approval required before execution. Pass approval_nonce as human_acknowledgment_token after obtaining human sign-off.",
  ...
}
```

#### Second call (with token)

Input:
```json
{
  "from_schema": "...",
  "to_schema": "...",
  "human_acknowledgment_token": "a3f7e2c1b8d04591"
}
```

Output (approved):
```json
{
  "risk_tier": "CRITICAL",
  "has_data_loss": true,
  "decision_required": false,
  "proceed_ok": true,
  "approved_at": 1747612800,
  ...
}
```

Output (invalid/expired token):
```json
{
  "error_code": "INVALID_ACKNOWLEDGMENT_TOKEN",
  "error": "Token not recognized or expired. Re-run analysis to obtain a fresh approval_nonce.",
  "retryable": true
}
```

### 2.3 Bridge Enforcement

The bridge (not the binary) manages nonce state, since Zero v0.1.x has no state beyond filesystem:

```bash
# Bash associative array in bridge process:
declare -A _PENDING_APPROVALS  # nonce → "from_hash:to_hash:expires_at"

# On decision_required: true response from binary:
nonce=$(generate_nonce)  # 8 bytes from /dev/urandom, hex-encoded
from_hash=$(sha256sum <<< "$from_schema" | cut -c1-16)
to_hash=$(sha256sum <<< "$to_schema" | cut -c1-16)
expires_at=$(( SECONDS + 300 ))  # 5 min TTL
_PENDING_APPROVALS[$nonce]="${from_hash}:${to_hash}:${expires_at}"
# Inject approval_nonce + approval_prompt into forge_out before returning
```

On second call with `human_acknowledgment_token`:
```bash
# Verify nonce exists, from/to hashes match, not expired
# If valid: clear nonce from _PENDING_APPROVALS (single-use)
# Inject proceed_ok: true into response
```

### 2.4 Nonce Generation

```bash
generate_nonce() {
    dd if=/dev/urandom bs=8 count=1 2>/dev/null | od -An -tx1 | tr -d ' \n'
}
```

16 hex chars = 64 bits of entropy. Brute-force infeasible within the 5-minute TTL.

### 2.5 Schema Addition

New field in `forge_analyze` `input_schema`:
```json
{
  "human_acknowledgment_token": {
    "type": "string",
    "description": "Nonce from a prior approval_nonce response. Pass this to confirm human sign-off on a decision_required migration. Max 64 bytes.",
    "maxLength": 64
  }
}
```

New fields in `output_schema`:
```json
{
  "approval_nonce": {
    "type": "string",
    "description": "Present when decision_required: true. Pass as human_acknowledgment_token after obtaining human sign-off."
  },
  "approval_prompt": {
    "type": "string",
    "description": "Human-readable instruction for the agent to surface to its principal."
  },
  "proceed_ok": {
    "type": "boolean",
    "description": "True when a valid human_acknowledgment_token was provided and verified. Agent may proceed."
  }
}
```

New error codes:
```json
{
  "INVALID_ACKNOWLEDGMENT_TOKEN": {
    "retryable": true,
    "fix": "Re-run forge_analyze (no token) to get a fresh approval_nonce"
  }
}
```

## 3. V34 Fix: Native Zero MCP Server

When Zero v0.1.x gains `world.in` (stdin reading), the bridge shell script is replaced by a Zero binary that directly implements MCP JSON-RPC 2.0 over stdio.

### 3.1 Architecture change

```
v0.1.x:  MCP client → mcp-bridge.sh (bash) → forge binary (Zero ELF64)
v0.2.x:  MCP client → forge binary (Zero ELF64, MCP native)
```

### 3.2 Required Zero stdlib additions

| API | Purpose |
|-----|---------|
| `world.in.readLine()` | Read one JSON-RPC request line |
| `std.rand.entropyBytes(n)` | Nonce generation for approval tokens |
| `std.json.parse(span)` | Structured JSON parsing (avoids manual field extraction) |
| `std.json.stringify(value)` | JSON serialization |

### 3.3 Migration path

The bash bridge remains authoritative until all four stdlib APIs land. The bridge's `bridge_note` field in `mcp-manifest.json` documents this explicitly.

## 4. Rate Limit State Persistence (RT-70c)

In v0.1.x, rate limit state is in-process bash arrays — resets on bridge restart, not shared across multiple bridge instances.

In v0.2.x (native Zero server), rate limit state must survive:
- Process restart: store buckets in `.forge-rate-state` (JSON, flock-protected)
- Concurrent connections: Zero's eventual threading model or a single-process design

### 4.1 Disk format (.forge-rate-state)

```json
{
  "buckets": {
    "forge_analyze:ip:session": {"tokens": 18400, "last_epoch": 1747612800},
    "forge_analyze_org:org:session": {"tokens": 48200, "last_epoch": 1747612800}
  },
  "schema_version": 1
}
```

### 4.2 Write protocol

1. Read current state with `std.fs.open` + `std.fs.read`
2. Compute new token values
3. Write to `.forge-rate-state.tmp` with `std.fs.createOrRaise`
4. Atomically rename: `std.fs.rename(".forge-rate-state.tmp", ".forge-rate-state")`

Step 4 requires `std.fs.rename` — blocked until Zero v0.2.

## 5. Open Questions

1. **Nonce lifetime**: 5 minutes may be too short for async human review workflows. Operator override env var `FORGE_APPROVAL_TTL_SECONDS`?
2. **Token scope**: Does the token bind to the exact (from, to) pair, or just to a risk tier? Binding to hashes prevents token reuse across different migrations.
3. **Audit log**: Should approved migrations be appended to `.forge-audit.jsonl` for compliance?
4. **Multi-instance**: If two forge bridges share a working directory, both have separate `_PENDING_APPROVALS` in memory. Tokens generated by instance A won't be recognized by instance B. Same problem as RT-70c. Disk-based nonce store (with the same flock protocol as rate limits) would fix this.
